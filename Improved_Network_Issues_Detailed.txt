### LAT + RPKT (High delay + many retransmitted packets)

**What's going wrong:**
Network congestion or buffer overflow at intermediate hops. Queue buildup increases latency while full buffers drop packets requiring retransmission. May also indicate interface speed mismatch or oversubscribed uplinks.

**Where to find the issue:**
- Check switch/router interface statistics (show interface counters, queue depth)
- Monitor bandwidth utilization on core/distribution layer switches (SNMP, NetFlow)
- Review QoS policies and queue configurations
- Check for jumbo frame mismatches between segments
- Examine buffer allocation on switches handling this traffic
- Look for egress queue drops in interface statistics

**How to fix:**
1. Identify congested links via SNMP monitoring (>80% utilization sustained)
2. Implement or adjust QoS policies to prioritize critical traffic
3. Upgrade link capacity on oversubscribed segments
4. Enable flow control (IEEE 802.3x) if appropriate for environment
5. Adjust buffer allocation on switches/routers
6. Consider traffic shaping to smooth bursts
7. Deploy LACP link aggregation for additional bandwidth
8. Segment traffic using VLANs to isolate heavy consumers

### LAT + R% (High delay + high loss percentage)

**What's going wrong:**
Severe congestion or device overload. Egress queues are consistently full, causing both queueing delay and tail drops. Could also be a failing interface, duplex mismatch, or critically overloaded intermediate device (switch/router CPU, memory exhaustion).

**Where to find the issue:**
- Check interface error counters (CRC errors, output drops, collisions, runts, giants)
- Verify duplex/speed settings on both ends (auto vs. hardcoded mismatches)
- Monitor device CPU and memory utilization during problem periods
- Review interface queue drop statistics (tail drops vs. WRED drops)
- Check for hardware errors in switch/router logs
- Verify cable integrity and port signal levels (for fiber, check dBm light levels)
- Check interface flapping history

**How to fix:**
1. Verify and hardcode duplex/speed settings on both ends (eliminate auto-negotiation issues)
2. Replace faulty cables or transceivers if physical errors detected
3. Upgrade device resources or redistribute traffic if device CPU/memory maxed out
4. Increase queue sizes where appropriate (within hardware limits)
5. Implement Weighted Random Early Detection (WRED) to reduce tail drops
6. Add parallel links and enable LACP for bandwidth aggregation
7. Segment traffic using VLANs to isolate heavy users
8. Replace aging hardware that cannot handle current traffic volumes

### LAT + PQF (High delay + sessions flagged poor)

**What's going wrong:**
Excessive queuing delay or jitter impacting application-layer performance. TCP connections remain established but experience degraded throughput and responsiveness. Often caused by bursty traffic patterns, inadequate QoS, or slow server/application responses compounding network delays.

**Where to find the issue:**
- Analyze jitter statistics on monitoring platforms
- Check application server response times (APM tools, application logs)
- Review queue depths and queueing algorithms (FIFO vs. WFQ vs. PQ)
- Monitor packet inter-arrival variance
- Check for microburst detection in switch analytics
- Review server CPU/memory/disk I/O during poor quality periods
- Correlate network delays with application transaction times

**How to fix:**
1. Implement priority queuing for latency-sensitive applications (VoIP, video, real-time apps)
2. Configure traffic shaping to reduce microbursts
3. Tune TCP window sizes on servers for better throughput
4. Enable Low Latency Queuing (LLQ) for priority traffic
5. Optimize application server performance (tune thread pools, connection pools, GC)
6. Consider separating latency-sensitive traffic to dedicated VLANs with stricter QoS
7. Implement application-level caching to reduce backend delays
8. Add packet buffering capacity to handle short-term bursts

### LAT + TO (High delay + timeouts)

**What's going wrong:**
Packets are delayed beyond TCP Retransmission Timeout (RTO) values (typically 200ms-3s depending on configuration). This forces TCP to assume packet loss and trigger expensive retransmission timeouts. Can be caused by extreme congestion, device CPU overload freezing packet forwarding, or processing delays in security appliances (firewall deep packet inspection, IPS scanning).

**Where to find the issue:**
- Check maximum queue depths and peak latency values
- Monitor firewall/IPS processing delays and session table utilization
- Review device CPU spikes correlating with timeout periods
- Check for routing loops or suboptimal path selection (traceroute analysis)
- Examine security appliance logs for dropped/delayed sessions
- Monitor application server backend database query times
- Check for DNS resolution delays

**How to fix:**
1. Offload firewall/IPS to more powerful hardware or distribute load across multiple devices
2. Tune firewall session timeout values appropriately for applications
3. Optimize security policy rules (move frequently hit rules to top, remove unnecessary deep inspection)
4. Increase device resources or optimize control plane protection
5. Implement bypass mechanisms for trusted internal traffic
6. Adjust TCP RTO parameters on servers if appropriate for environment (careful tuning required)
7. Fix routing issues to prevent suboptimal paths (adjust metrics, fix routing protocols)
8. Optimize slow database queries causing application-layer delays
9. Enable hardware acceleration for crypto/inspection functions

### LAT + UNI (High delay + one-way traffic)

**What's going wrong:**
Asymmetric routing with return path failure or severe impairment. Forward packets arrive (slowly) but acknowledgments don't return properly, or return via a broken/severely congested path. Common causes include: stateful firewall session timeout, NAT state table exhaustion, asymmetric routing through different firewall contexts, or return path connection failures.

**Where to find the issue:**
- Use traceroute/tracepath in BOTH directions to identify asymmetry
- Check firewall state tables for session exhaustion (show conn count)
- Review NAT translation tables and timeout settings
- Examine routing tables for asymmetric paths (different ingress/egress)
- Check firewall logs for dropped return packets (stateful inspection failures)
- Verify spanning tree topology hasn't changed unexpectedly
- Review load balancer hash algorithm causing flow splits
- Check for ECMP hash polarization

**How to fix:**
1. Implement symmetric routing using policy-based routing (PBR)
2. Increase firewall state table sizes and adjust timeout values
3. Configure connection tracking synchronization between redundant firewalls
4. Ensure both directions traverse the same stateful devices
5. Fix NAT pool exhaustion by expanding address pools
6. Adjust load balancer hashing algorithms to maintain flow symmetry (5-tuple hashing)
7. Enable TCP timestamp options for better PAWS (Protection Against Wrapped Sequences) handling
8. Consider source-based routing to force path symmetry
9. Fix spanning tree to ensure consistent L2 paths
10. Implement stateful HA synchronization properly

### RPKT + R% (High retrans volume + high retrans rate)

**What's going wrong:**
Significant real packet loss on the wire. Root causes: physical layer problems (bad cable, dirty/damaged fiber, SFP transceiver issues), duplex mismatch (very common culprit showing as late collisions), congestion drops from sustained oversubscription, or faulty network interface card (NIC).

**Where to find the issue:**
- Check interface error counters (input/output errors, CRC, runts, giants, fragments)
- Verify duplex and speed settings match EXACTLY on both ends
- Test cable integrity with cable testers (TDR for copper, OTDR for fiber)
- Check optical power levels (dBm) on fiber links (both TX and RX)
- Review interface statistics for late collisions (classic duplex mismatch indicator)
- Monitor NIC driver error logs on servers (/var/log/messages, Event Viewer)
- Check switch port LED indicators for anomalies (error blinks)
- Review environmental factors (temperature, power)

**How to fix:**
1. Replace faulty cables immediately (primary cause of physical errors - 70%+ of issues)
2. Clean fiber connectors with proper cleaning tools and replace if damaged
3. Hardcode duplex/speed settings to eliminate auto-negotiation issues (manual config on both sides)
4. Replace faulty NICs or update to latest stable drivers
5. Replace bad SFP/QSFP modules (verify compatibility with switch)
6. Upgrade oversubscribed links immediately
7. Check and eliminate sources of EMI interference for copper (fluorescent lights, motors, power cables)
8. Verify proper grounding of all network equipment
9. Check for cable bend radius violations
10. Replace aging cables preemptively (copper >7 years, fiber if showing light level degradation)

### RPKT + TO (Many retransmissions + timeouts)

**What's going wrong:**
Sustained or high-frequency packet loss overwhelming TCP's fast retransmit/recovery mechanisms. Normal fast retransmit (3 duplicate ACKs) fails to recover packets, triggering expensive timeout-based recovery and exponential backoff. Indicates severe congestion, actively failing link, or critical device malfunction.

**Where to find the issue:**
- Monitor packet loss percentage over sliding time windows
- Check for interface flapping in device logs
- Review device health indicators (fans, temperature, PSU status)
- Check for hardware errors in device diagnostics (ASIC errors, memory errors)
- Monitor for sustained high utilization periods (>90% for extended time)
- Review spanning tree convergence events (topology changes)
- Check for broadcast/multicast storms or L2 loops
- Look for link errors correlating with timeout events

**How to fix:**
1. **Emergency action**: Temporarily reduce traffic load (block non-critical sources, rate-limit)
2. Identify and replace failing hardware immediately (cables, transceivers, NICs, switches)
3. Fix spanning tree issues to prevent loops (BPDU guard, root guard, loop guard)
4. Implement storm control to limit broadcast/multicast impact
5. Increase link capacity urgently if sustained oversubscription confirmed
6. Redistribute traffic across multiple paths (enable ECMP, LACP)
7. Implement fast convergence protocols (RSTP instead of STP, BFD for fast failure detection)
8. Add redundant links for critical paths with fast failover
9. Deploy link monitoring (dying gasp detection, BFD)
10. Implement graceful degradation mechanisms
